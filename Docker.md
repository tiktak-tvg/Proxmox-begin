### Работа с докером 

1. Установить Docker на хост (помня о рисках)

2. Обновить систему и потом установить Docker в VM

3. Просто обновить пакеты Proxmox, не трогая Docker

#### Три варианта действий: подробное описание

#### Вариант 1. Установить Docker на хост (помня о рисках)
Что это значит: Docker устанавливается непосредственно на операционную систему Proxmox VE (Debian 11 Bullseye), которая является гипервизором. 

Все контейнеры Docker будут работать на том же уровне, что и LXC-контейнеры и KVM-виртуальные машины.

##### Шаги (выполнены частично)
Обновление системы — вы уже сделали:
```bash
apt update && apt upgrade -y
apt dist-upgrade -y   # для полного обновления, включая ядро
reboot
```
Добавление официального репозитория Docker (выполнено):
```bash
mkdir -p /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/debian/gpg -o /etc/apt/keyrings/docker.asc
chmod a+r /etc/apt/keyrings/docker.asc
echo "deb [arch=amd64 signed-by=/etc/apt/keyrings/docker.asc] https://download.docker.com/linux/debian bullseye stable" > /etc/apt/sources.list.d/docker.list
apt update
```
Установка Docker (ещё не сделано, можно выполнить):
```bash
apt install docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin -y
```
Проверка:
```bash
systemctl enable docker && systemctl start docker
docker run hello-world
```
### Основные риски
- **Конфликт сетевых правил**

Docker изменяет ``iptables`` (добавляет свои цепочки DOCKER). Это может нарушить работу LXC-контейнеров и моста vmbr0, который Proxmox использует для связи VM и контейнеров с внешней сетью. Симптомы: пропадает доступ к контейнерам, не работает NAT, не поднимаются бриджи.

Изменение настроек ``nftables`` (если используется)

Современные версии Debian могут переходить на ``nftables``. Docker пока не полностью совместим с ним.

- **Конфликт с firewalld**

Proxmox обычно не использует ``firewalld``, но если вы его установите – Docker может нарушить его работу.

- **Проблемы с AppArmor/SELinux**

Docker и LXC могут бороться за профили безопасности.

- **Сложность отладки**

Когда сеть перестаёт работать, трудно понять, кто виноват – ``Docker или Proxmox``. Часто помогает только полное отключение Docker.

**Когда этот вариант оправдан**
Тестовая или домашняя среда, где потеря нескольких контейнеров не критична.

Вы не используете LXC-контейнеры (только KVM) – с KVM конфликтов меньше.

Готовы перезагружать сервер и отключать Docker в случае проблем.

**Когда не стоит так делать**
Продакшен-среда.

Активное использование LXC.

Нужна гарантированная стабильность сети.

#### Минимизация рисков
После установки Docker проверьте LXC-контейнеры: создайте тестовый, убедитесь, что он получает IP и пингуется.

Отключите автоматическое изменение ``iptables Docker: в /etc/docker/daemon.json добавьте { "iptables": false }``. Но тогда не будет проброса портов -p.

Настройте Docker использовать существующий мост ``vmbr0`` вместо создания ``docker0``.

### Вариант 2. Обновить систему и потом установить Docker в виртуальную машину (VM)
Что это значит: Вы создаёте полноценную виртуальную машину (KVM) внутри Proxmox, на неё ставите Debian/Ubuntu, а уже внутри неё – Docker. Хост Proxmox остаётся чистым.

Это рекомендованный и безопасный способ.

##### Шаги
- Обновить хост Proxmox – вы уже сделали:
```bash
apt update && apt upgrade -y && apt dist-upgrade -y
reboot
```
- Создать виртуальную машину через веб-интерфейс Proxmox:

    - Нажмите Create VM

    - Общие: ID (например, 100), имя docker-host

    - ОС: выберите образ Debian 12 или Ubuntu 22.04/24.04 (ISO загрузите в хранилище local)

    - Система: BIOS по умолчанию (SeaBIOS), SCSI контроллер VirtIO SCSI

    - Диск: 20-30 GB, кэш Write back

    - CPU: 2-4 ядра (зависит от нагрузки)
    
    - Память: минимум 2 GB, лучше 4 GB

    - Сеть: мост vmbr0, модель VirtIO (лучше производительность)

    - Подтвердить и запустить VM

- Установить ОС внутри VM (обычная установка через VNC-консоль)

- Настроить сеть внутри VM (обычно DHCP, но лучше статический IP)

- Установить Docker внутри VM (стандартные команды для Debian/Ubuntu):
```bash
apt update && apt install curl -y
curl -fsSL https://get.docker.com -o get-docker.sh
sh get-docker.sh
usermod -aG docker $USER
```
- Проверить:
```bash
docker run hello-world
```
##### Преимущества
- Полная изоляция: проблемы Docker не влияют на хост и другие VM/контейнеры.

- Можно делать снапшоты VM перед экспериментами.

- Удобное управление ресурсами: лимиты CPU/RAM для Docker-хоста.

- Легко мигрировать на другой узел Proxmox.

- Безопаснее: если Docker-контейнер скомпрометирован, злоумышленник окажется внутри VM, а не на хосте.

##### Недостатки
- Небольшое падение производительности (обычно менее 5-10%).

- Нужно выделить отдельные ресурсы под VM (память, диск, CPU).

- Нет прямого доступа к железу хоста (например, к GPU без проброса).

##### Дополнительные советы
Для более лёгкого решения вместо KVM можно использовать LXC-контейнер (привилегированный, с включённой вложенностью), но безопасность будет ниже, чем у VM.

Настройте автоматический старт VM после загрузки хоста.
